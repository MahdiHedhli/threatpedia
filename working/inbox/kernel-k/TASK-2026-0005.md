## Task

Modify the GitHub workflow rules to allow PR-less commits to `working/` from authorized agent accounts (dangermouse-bot, eventually penfold-bot), while maintaining PR protection for the live corpus (`incidents/`, `campaigns/`, `entities/`, `docs/`, `.github/`).

## Context

The working directory protocol assumes agents can commit directly to `working/` so that the inbox/outbox/log cycle works without manual intervention from you. The current workflow on threatpedia requires PRs for all commits, which would force you to approve every single agent action — exactly the bottleneck the working directory is designed to escape.

## Acceptance Criteria

- [ ] Audit current `.github/workflows/*.yml` to understand existing CI rules
- [ ] Add or modify branch protection rules on `main` to:
  - Allow direct push from `dangermouse-bot` and `penfold-bot` accounts to paths under `working/`
  - Continue requiring PRs (or signed commits with review) for paths under `incidents/`, `campaigns/`, `entities/`, `docs/`, `assets/`, and `.github/`
- [ ] Add a CODEOWNERS file (or update existing) that enforces review on protected paths
- [ ] Optional: add a webhook or workflow that notifies you on Slack/email for every agent commit
- [ ] Optional: add a CI check that fails if a commit by an agent account touches a protected path
- [ ] Test by having DM (or an actual agent runtime) make a commit to `working/` and verify it lands without PR

## Suggested approach

GitHub branch protection rules don't natively support per-path restrictions. Two ways to enforce this:

### Option 1 — CODEOWNERS + required review
```
# .github/CODEOWNERS
/incidents/ @MahdiHedhli
/campaigns/ @MahdiHedhli
/entities/ @MahdiHedhli
/docs/ @MahdiHedhli
/assets/ @MahdiHedhli
/.github/ @MahdiHedhli

# working/ has no owner — agents can commit freely
```
Combined with branch protection requiring CODEOWNERS approval, agents can push to `working/` without review but cannot push to protected paths without your approval.

### Option 2 — Pre-receive hook via GitHub Action
A workflow that runs on push and rejects the push if a non-Kernel-K commit touches protected paths. More complex but more flexible.

DM recommends Option 1 — CODEOWNERS is built into GitHub, well-understood, and zero custom code.

## Notes

This is P1 because nothing fundamentally breaks without it — Kernel K can manually review and approve every working/ commit in the short term. But the protocol is designed to scale beyond that, and the workflow update is the prerequisite.

Until this lands, every commit by DM (and eventually Penfold) goes through Kernel K. After it lands, only commits to the live corpus need review.
