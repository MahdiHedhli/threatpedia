## Task

Decide which runtime architecture will host the autonomous dangermouse-bot worker.

## Context

The working directory protocol assumes that some runtime, somewhere, is able to commit to the repo as `dangermouse-bot` autonomously. DM-in-Chat (the Claude instance you talk to) cannot do this — it has no persistent identity, no scheduled execution, and no token storage. The runtime is a separate thing that needs to exist.

Per your constraint of "no API tax until the project is making money," API-based options (Option 2 in DM's previous analysis: GitHub Actions calling Anthropic API) are off the table for now.

## Acceptance Criteria

- [ ] Choose a runtime from the options below (or propose a new one)
- [ ] Document the choice as ADR 0004 in `working/decisions/`
- [ ] Note any prerequisites (token storage, machine availability, GPG key generation, etc.)
- [ ] Set a target date for the runtime going live

## Options to consider

### Option A — Claude Code on a long-running machine
- **Where:** Your dev box, a Hetzner VPS, or a Mac mini at home
- **Token storage:** `~/.config/gh/hosts.yml` (set by `gh auth login --with-token`)
- **Trigger:** You run it manually, OR a cron wraps it to poll `working/inbox/dangermouse/` periodically
- **Pros:** Lowest friction since you already use Claude Code; familiar tooling
- **Cons:** Requires a machine that's always on (or scheduled); manual trigger pattern doesn't fully solve the "you as bottleneck" problem until cron is set up
- **API tax:** Claude Code is included with your Anthropic subscription, no extra API charges

### Option C — Antigravity (when freeze lifts)
- **Where:** Their cloud
- **Token storage:** Their secrets vault
- **Trigger:** Their scheduler
- **Pros:** Built for this use case; you've already paid for it; their vault is purpose-built for credential storage
- **Cons:** You paused them during the freeze; they were inconsistent in earlier testing; some lock-in
- **API tax:** Subscription you're already paying

### Option D — Custom wrapper on a VPS
- **Where:** Hetzner / DigitalOcean / your own infra
- **Token storage:** systemd credentials, or HashiCorp Vault, or doppler.com
- **Trigger:** systemd timer or cron
- **Pros:** Full control; minimal external dependencies
- **Cons:** You build and maintain it; more code surface to secure
- **API tax:** None for the runtime itself, plus VPS hosting cost (~$5-20/mo)

### Option E — Cowork (your existing tool)
- **Where:** Desktop session
- **Token storage:** Local OS keychain
- **Trigger:** You start a session manually
- **Pros:** Already paid for; integrated with desktop
- **Cons:** Inconsistent between sessions per your earlier feedback; manual trigger; not autonomous
- **API tax:** Subscription you're already paying

## DM's recommendation

**Short-term (this week):** Option A — Claude Code on your dev box. You're already using Claude Code. Set up a clone of the repo with the dangermouse-bot token in `gh`. Run it manually for now to process the inbox. Zero new infrastructure.

**Medium-term (when freeze lifts):** Option C — Antigravity for scheduled jobs (nightly archiving, weekly audits, monthly reports). The scheduling and credential vault are exactly what Antigravity is for.

**Long-term (post-launch):** Option D — Custom wrapper if/when Antigravity becomes a constraint. By then we'll know what the actual workload looks like and can spec it correctly.

This three-stage migration lets you pay zero new costs in the short term, layer on Antigravity when you're ready, and reserve custom infra for when there's a clear need.

## Notes

This is P1 because the working directory protocol does work without a runtime — you're acting as the runtime right now, manually committing on DM's behalf. The runtime is a quality-of-life upgrade that removes you from routine commits, not a blocker.
