# Threatpedia Git Safety Protocol

**All scheduled agents MUST follow this protocol for every git commit/push operation.**

This protocol prevents lock file conflicts and merge failures when multiple agents operate on the same repository concurrently.

## Before Any Git Work

```bash
cd /Users/mhedhli/Documents/Threatpedia/threatpedia

# 0. Configure bot identity and auth (credential helper keeps PAT out of .git/config)
BOT_PAT=$(cat .env | tr -d '[:space:]')
git config user.name "Danger Mouse"
git config user.email "dangermousebot@threatpedia.wiki"
git remote set-url origin "https://github.com/MahdiHedhli/threatpedia.git"
git config credential.helper "!f() { echo \"username=dangermouse-bot\"; echo \"password=${BOT_PAT}\"; }; f"

# 1. Clear stale lock files (>5 min old = stale; another task likely crashed)
if [ -f .git/index.lock ]; then
  lock_age=$(( $(date +%s) - $(stat -f %m .git/index.lock 2>/dev/null || stat -c %Y .git/index.lock 2>/dev/null) ))
  if [ "$lock_age" -gt 300 ]; then
    rm -f .git/index.lock
    echo "LOCK: Cleared stale index.lock (${lock_age}s old)"
  else
    echo "LOCK: index.lock is recent (${lock_age}s old) — waiting 90s for other task to finish..."
    sleep 90
    if [ -f .git/index.lock ]; then
      rm -f .git/index.lock
      echo "LOCK: Cleared index.lock after waiting"
    fi
  fi
fi

# Also clear other potential lock files
rm -f .git/refs/heads/main.lock 2>/dev/null
rm -f .git/HEAD.lock 2>/dev/null

# 2. Fetch and rebase to latest remote state
git fetch origin main 2>&1
git rebase origin/main 2>&1
if [ $? -ne 0 ]; then
  echo "REBASE: Conflict detected — aborting rebase and trying merge..."
  git rebase --abort 2>&1
  git merge origin/main --no-edit 2>&1
  if [ $? -ne 0 ]; then
    echo "MERGE: Conflict detected — accepting remote changes for JSON files..."
    git checkout --theirs incidents/manifest.json task-state.json glossary-index.json glossary-candidates.json 2>/dev/null
    git add -A 2>&1
    git commit --no-edit 2>&1 || true
  fi
fi
```

## After Staging Changes — Before Commit

```bash
# 3. Check lock again (may have appeared while working)
rm -f .git/index.lock 2>/dev/null

# 4. Stage specific files (never use git add -A or git add .)
git add [list specific files here]
```

## Commit and Push with Retry

```bash
# 5. Commit
git commit -m "message here"

# 6. Push with pull-retry loop (max 3 attempts)
push_success=false
for attempt in 1 2 3; do
  git push origin main 2>&1
  if [ $? -eq 0 ]; then
    push_success=true
    break
  fi
  echo "PUSH: Attempt $attempt failed — pulling and retrying..."
  rm -f .git/index.lock 2>/dev/null
  git pull --rebase origin main 2>&1
  if [ $? -ne 0 ]; then
    git rebase --abort 2>&1
    git pull origin main --no-edit 2>&1
    # For JSON coordination files, accept remote on conflict
    git checkout --theirs incidents/manifest.json task-state.json 2>/dev/null
    git add -A && git commit --no-edit 2>&1 || true
  fi
done

if [ ! "$push_success" = true ]; then
  echo "ERROR: Push failed after 3 attempts. Logging for manual intervention."
fi
```

## JSON Conflict Strategy

For coordination files (`manifest.json`, `task-state.json`, `glossary-index.json`, `glossary-candidates.json`):
- On conflict, accept REMOTE version (`--theirs`) then RE-APPLY your changes on top
- Read the remote version, merge your additions into it, and write the result
- This prevents data loss from concurrent updates

## Key Rules

1. **Never use `git add -A` or `git add .`** — always stage specific files by name
2. **Always fetch/rebase BEFORE starting work** — not just before pushing
3. **Check for lock files at EVERY git operation** — not just at the start
4. **For JSON files, prefer merge over overwrite** — read remote, merge your changes, write result
5. **If push fails 3 times, STOP** — log the error and skip the push; a human or the next run will resolve it
6. **Never force push** — `git push --force` is never acceptable
7. **Always configure bot identity** — run step 0 (git config user.name/email + credential helper) before any git operation
8. **PRs are opened by the bot** — when creating PRs via the GitHub API, use the bot PAT from `.env` so the PR is attributed to `dangermouse-bot`, enabling MahdiHedhli to review and approve

## Gemini Code Assist Review Protocol

All tasks that create PRs **MUST** check for and handle Gemini Code Assist review
comments. See `.github/GEMINI-REVIEW-PROTOCOL.md` for the full protocol.

Use the shared handler at `scrapers/gemini_review.py`:

```python
from gemini_review import GeminiReviewHandler

handler = GeminiReviewHandler(token_path="path/to/.envbot")

# Pre-check: before main work, scan all open PRs
results = handler.precheck_all_open_prs()
for pr_num, result in results.items():
    for comment in result.actionable:  # critical + high
        # Fix the issue, then reply:
        handler.reply_to_comment(pr_num, comment.comment_id,
                                  "Brief fix description", "commit_hash",
                                  comment.comment_type)

# Post-check: after creating a PR, monitor for comments
result = handler.postcheck_pr(pr_number=26, wait_minutes=10)
for comment in result.all_comments:
    handler.reply_to_comment(26, comment.comment_id,
                              "Fix or acknowledgment", "commit_hash",
                              comment.comment_type)
```

### Severity Triage

| Severity | Action |
|----------|--------|
| Critical | Fix immediately (clone → checkout branch → fix → commit → push → reply) |
| High | Fix immediately (same as critical) |
| Medium | Fix if < 5 min; otherwise acknowledge with reply |
| Low | Acknowledge with reply, defer to next run |

### Reply Format

```
Fixed in {commit_hash} — {description}
```

Always add an `eyes` reaction to each Gemini comment to signal it has been reviewed.
